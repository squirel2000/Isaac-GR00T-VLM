1 · 模型關係：獨立 VLM、VLA、LoRA 合併與 swap

本專案不修改 GR00T N1.7 的訓練流程，而是「獨立載入完整 Cosmos-R2 → LoRA SFT → 合併（產物 A）→ 把 backbone.model.* 權重置換回 VLA」。下圖為兩條獨立堆疊：上方是可單獨部署的 VLM，下方是 GR00T VLA（backbone + action head）。

依目前 checkout 的程式碼與 checkpoint 實測：checkpoint-150000/config.json 為 select_layer=16、vl_self_attention_cfg.num_layers=4、diffusion_model_cfg.num_layers=32；VLA safetensors 內 backbone.model.* 493 個 tensor、action_head.* 537 個（含 vlln 2、vl_self_attention 64），VLA 的 LLM 實際只有 layers 0..15。合併後的產物 A 是單一 model.safetensors 的 Qwen3VLForConditionalGeneration，LLM layers 0..27。各名詞與層數歸屬詳見附錄 A。

2 · LLM 微調：方法總覽

方法更新什麼成本 / 顯存適用
Full fine-tune全部權重最高（需存全部梯度 + 優化器狀態）資料多、要最大改變
LoRA凍結原權重，學低秩旁路 A·B低（僅 ~0.1–1% 參數）多數領域微調（本專案）
QLoRA4-bit 量化底座 + LoRA最低（單卡可調大模型）顯存受限
Adapter每層插入小瓶頸模組低多任務切換
Prompt / Prefix tuning只學軟提示向量極低輕量任務適配

LoRA 原理（動畫）：凍結 W，只學低秩 A·B

標準步驟

  1. 資料準備：整理成指令／對話格式；切 train/val；定義標籤遮罩（只對「答案」算 loss）。
  2. 選 target 模組：LoRA 通常掛在注意力 q/k/v/o 與 MLP gate/up/down proj。
  3. 超參數：r（秩）、α（縮放）、dropout、lr（1e-4 常見）、batch × grad-accum、warmup、steps。
  4. 訓練：bf16 + gradient checkpointing 省顯存；監看 loss 與 grad_norm。
  5. 評估：held-out 準確率／任務指標；避免過擬合（小資料用小 r + 早停）。

5 · Dense vs MoE（非 dense）

MoE 路由（動畫）：每 token 只啟動少數 expert

MoE 機制：router、expert、協同運作

組件 / 面向說明
Router（gating）每個 MoE 層內建一個 gating 線性層，對每個 token 算出對各 expert 的分數（softmax）、選出 top-k；與 expert 一起訓練。
Expert每個 expert 是一個獨立前饋網路（FFN）；同層有 N 個並行 expert，訓練中自然特化到不同輸入模式／領域。
協同運作router 為每個 token 選 top-k → 只計算那 k 個 → 用 router 分數加權合併其輸出；其餘 expert 對該 token 不計算（省算力的來源）。
容量 / 成本固定算力下提供更大「總容量」。以 Qwen3-VL-30B-A3B 為例：31B 總參數、每 token ~3B 啟動 → 推論快、表達力強；代價是所有 expert 權重都須進顯存（~62 GB），且需 Qwen3VLMoeForConditionalGeneration 類別。dense 8B 則每 token 走全部參數、~17 GB、載入單純。

附錄 A · 技術問答（Q&A）

以下為與正文互補的深入技術問答，集中於名詞歸屬、流程通用性、MoE 觀察與訓練目標的差異；正文已說明的概念此處不重述，只補細節。

A-1 · 架構與名詞歸屬

Q1 vlln 是什麼？vl_self_attention 算 VLM 還是 DiT？
vlln 是 Vision-Language LayerNorm 的專案命名：在 Gr00tN1d7ActionHead 中實作為 nn.LayerNorm(backbone_embedding_dim)，本 checkpoint 是 LayerNorm(2048)；它屬 action_head.vlln.*，不是 VLM backbone。vl_self_attention 也不是 backbone、也不是 DiT 本體，而是 action head 內、接在 vlln 後的 4 層 VL feature adapter（action_head.vl_self_attention.*）。DiT 本體在 action_head.model.*，本 checkpoint 為 32 層。
Q2 Qwen3VLForConditionalGeneration 是 transformers 公定格式嗎？
不是跨所有 VLM 的公定類別名稱，而是 HF Transformers 為 Qwen3-VL 架構提供的模型 class。公定／通用的是介面慣例：PreTrainedModel、from_pretrained()、save_pretrained()、generate()、AutoProcessor、以及較通用的 AutoModelForImageTextToText。換成 LLaVA、Idefics、Gemma、Mllama 等，會換成各家族自己的 class 或對應 AutoModel。
Q3 為什麼程式碼預設 select_layer=12，但 checkpoint 是 16？
configs/model/gr00t_n1d7.py 的 12 只是 dataclass fallback；訓練／儲存時 config 可覆寫它。本例 checkpoint-150000/config.json 明寫 select_layer=16，safetensors 也只見 ...language_model.layers.0..15。「程式預設 12」不等於「所有產物都用 12」；部署以各 checkpoint 的 config.json 為準。

A-2 · 合併、置換與特徵偏移

Q4 為什麼第 16–27 層的 LoRA delta 被丟棄？不能整個 28 層放回去嗎？
獨立 LoRA 確實在完整 28 層 VLM 上訓練，merged VLM 有 layers 0..27。但 GR00T N1.7 VLA backbone 初始化時執行 while len(language_model.layers) > select_layer: pop(-1)，本例 select_layer=16，故 VLA 只存在 layers 0..15 的目的 key。swap_backbone.py 以「VLA key 為主」逐一找 merged 對應 tensor；VLA 沒有 16–27 層的 key，就沒地方可寫。硬塞 28 層會改變結構、forward 計算量與 hidden 分佈，也會讓已訓練的 action head 接到不同層語意 → 那是重建／重訓 VLA，而非 swap。
Q5 為什麼需要 merge？「底座」是什麼？
LoRA 輸出的 adapter_model.safetensors 只存低秩 delta（A/B），不是完整 VLM。底座＝原始 pretrained base（如 nvidia/Cosmos-Reason2-2B）。未 merge 時推論需「底座 + adapter」一起載入；merge 會把 ΔW = B·A·scale 寫回底座，輸出完整 standalone VLM（產物 A），可用普通 from_pretrained() 載入、也可作為 swap 來源。PEFT 的 merge_and_unload 即為此用途。
Q6 .safetensors 就是「完整 VLM 參數檔」嗎？swap/merge 是針對此檔案置換嗎？
.safetensors 只是安全的 tensor 儲存格式，要看目錄與 key：cosmos_r2_lora_*/checkpoint-*/adapter_model.safetensors 是 LoRA adapter（非完整 VLM）；GR00T VLA sharded checkpoint 含 backbone.model.* 與 action_head.*；merged 產物 A 的 model.safetensors 才是完整 standalone VLM。merge_lora.py 載入底座 + adapter 輸出 merged；swap_backbone.py 只替換 VLA 中形狀相符的 backbone.model.*。
Q7 「凍結 adapter/DiT 需承受小幅 VL 特徵偏移」具體是什麼偏移？
特徵張量形狀仍是 [B, seq_len, 2048]，但 swap 後 vision/embedding/LLM 前 16 層權重經 VQA LoRA 改過，每個 token 的 hidden vector 數值分佈會變（均值/方差、attention 後 token 關係、物體/顏色/任務文字 token 的語意方向、各層 feature scale）。由於 vlln、4 層 adapter、32 層 DiT、state/action 編解碼器都保持原權重，必須仍能把這批稍變的 VL features 映射到正確 action。偏移小 → action MSE/MAE 持平或改善；偏移大 → DiT cross-attention 讀到不同語意/尺度的條件 → 動作誤差上升。這就是 swap 後必跑 VLA action eval 的原因。

A-3 · VQA 資料與 teacher 觀察

Q8 為什麼 30B-A3B 的答案比 8B 簡短／籠統？是 MoE 的本質嗎？
非 MoE 本質。主因：(1) 兩模型 instruction-tuning 風格不同（30B 偏精簡）；(2) 都用 greedy 解碼（do_sample=False）；(3) 樣本小、屬軼事級觀察。不能推論「MoE 就比較籠統」。此外「8B 每影格固定 4 題」是 prompt 要求（「generate 4 diverse QA pairs」）而非模型限制——30B 有時會多給 3–7 題。
Q9 若缺 Summary / Temporal / Trajectory 題型會怎樣？
這三類分別測「場景整體理解／時間順序／動作路徑」。缺了，VQA 會偏向靜態屬性（顏色/位置），無法評估時序與全局推理；而時序與軌跡理解正是與 action 最相關的維度，缺了會低估 VLM 對操作任務的幫助。本專案 7 類齊全即為涵蓋這些面向。

A-4 · 訓練目標：CausalLM loss、Trainer 與動作維度

Q10 什麼是 CausalLM loss？和 HF Trainer 有何不同？
兩者是不同層次、非替代關係。CausalLM loss ＝ 自回歸「預測下一個 token」的交叉熵，是訓練目標：把 labels 傳進 Qwen3VLForConditionalGeneration 時模型內部即計算，本專案再加 label-mask 只對答案 token 計入。HF Trainer ＝ 訓練的流程引擎：負責 batching、grad-accum、optimizer step、lr schedule、logging、存檔、分散式，不決定 loss。兩者正交：Trainer 是外圈迴圈，CausalLM loss 是其中的目標函數。
Q11 GR00T 若也微調 backbone，訓練方式也不同嗎？
仍然不同。即使 GR00T 解凍 backbone（tune_llm=True / tune_top_llm_layers>0 / 對 backbone 掛 LoRA），它不是用 CausalLM／VQA loss、也不是 HF Trainer。backbone 的梯度是從 flow-matching action loss 經 DiT／adapter 反傳回來——目的是讓 backbone 產生「更利於預測動作的特徵」，而非更會生成文字。差異不只在「動不動 backbone」，而在目標函數與訓練迴圈本身：GR00T ＝ 自家 trainer + flow-matching action loss；本流程 ＝ HF Trainer + CausalLM 文字 loss。
Q12 為什麼 flow-matching action loss 是 26 維，不是 32？
26 是 OpenArm O6 這個 embodiment 的真實動作維度：雙臂各 7-DoF（14）＋雙靈巧手各 6-DoF（12）＝ 26（見 dataset 的 action.shape=[26]）。模型為支援多體系，把動作補零到 max_action_dim=132，並以 action_mask 只對真實 26 維算 loss。32 不是動作維度——那是 DiT 的 attention heads（32）與 max_num_embodiments（32）。「26」＝真實自由度、「32」＝注意力頭／體系數，兩者無關。
Q13 train_vlm_lora.py 是所有 VLM 通用的腳本嗎？其他架構怎麼做？
流程是標準 HF/PEFT SFT 配方（見 §3 對照表），但不是零修改通用：目前綁定 Qwen3-VL class，LoRA target 假設 Qwen 模組名（q/k/v/o/gate/up/down_proj）。同屬 Qwen3-VL / Cosmos-R2 相容架構通常可直接用；其他架構的標準範本是——換成該模型的 AutoModelForImageTextToText 或專屬 class、確認 processor/chat template、重選 target_modules、保持 label masking 與 Trainer 流程。即「流程標準、但 model class 與 target module 名稱依架構調整」。
