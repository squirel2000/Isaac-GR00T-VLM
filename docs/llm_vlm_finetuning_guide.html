LLM 與 VLM 微調指南

從一般大型語言模型（LLM）到視覺語言模型（VLM）的微調方法、挑戰與解法；以本專案（Cosmos-Reason2-2B VQA LoRA）為實例。含動畫資料流圖。

1 · LLM 微調：方法總覽

方法更新什麼成本/顯存適用
Full fine-tune全部權重最高（需存全部梯度+優化器狀態）資料多、要最大改變
LoRA凍結原權重，學低秩旁路 A·B低（僅 ~0.1–1% 參數）多數領域微調（本專案）
QLoRA4-bit 量化底座 + LoRA最低（單卡可調大模型）顯存受限
Adapter每層插入小瓶頸模組低多任務切換
Prompt / Prefix tuning只學軟提示向量極低輕量任務適配

LoRA 原理（動畫）：凍結 W，只學低秩 A·B

只有低秩 A、B（綠）可訓練，原始 W 凍結。合併時 W' = W + B·A，得到與原模型同形狀的權重 → 可直接「置換」回原架構（本專案 swap 的基礎）。本專案：r=16, α=32, 僅 0.81% 參數可訓練。

標準步驟

  1. 資料準備：整理成指令/對話格式；切 train/val；定義標籤遮罩（只對「答案」算 loss）。
  2. 選 target 模組：LoRA 通常掛在注意力 q/k/v/o 與 MLP gate/up/down proj。
  3. 超參數：r（秩）、α（縮放）、dropout、lr（1e-4 常見）、batch×grad-accum、warmup、steps。
  4. 訓練：bf16 + gradient checkpointing 省顯存；監看 loss 與 grad_norm。
  5. 評估：held-out 準確率／任務指標；避免過擬合（小資料用小 r + 早停）。

2 · VLM 微調：多模態的特定挑戰與解法

影像如何進入語言模型（動畫資料流）

挑戰解法（本專案採用）
多模態對齊：影像特徵 vs 文字 token 空間vision encoder → projector 投影到 LLM embedding；chat template 插入 <image> 佔位 → 被展開為 N 個 image token
image token 與特徵數必須相符content 用結構化 [{type:image},{type:text}]（非純字串），否則 token=0 vs features=N 報錯（本專案踩過此坑並修正）
vision encoder 要不要微調？多數 VQA LoRA 凍結 vision、只調 LLM 投影；省顯存又保留視覺泛化
loss 設計label masking：只對 assistant 答案 token 算 loss（prompt/影像 token 設 -100）
資料品質/多樣性teacher-VLM 自動生成 + template 高信度骨幹 + 揭露多樣性侷限

4 · Dense vs MoE（非 dense）

MoE 路由（動畫）：每 token 只啟動少數 expert

Router 為每個 token 挑選少數 expert（綠色啟動）。Qwen3-VL-30B-A3B：31B 總參數，每 token ~3B 啟動 → 算力像小模型、品質像大模型；但全部 expert 都要載進顯存（~62GB）。dense（8B）則每 token 走全部參數、~17GB、載入單純。

MoE 深入：router、expert、協同運作（你問的）

問題說明
router 是內建組件嗎？是。每個 MoE 層內建一個 gating(router) 線性層，對每個 token 算出對各 expert 的分數(softmax)，選出 top-k。它與 expert 一起被訓練。
expert 是什麼？每個 expert 就是一個獨立的前饋網路(FFN)；同一層有 N 個並行 expert。訓練中各 expert 會自然特化到不同的輸入模式/領域。
如何協同？router 為每個 token 選 top-k expert → 只計算那 k 個 → 用 router 分數加權合併它們的輸出；其餘 expert 對該 token 不計算（這就是省算力的來源）。
微調的作用/優勢固定算力下提供更大的「總容量」(31B)，但每 token 僅 ~3B 參與計算 → 推論快、表達力強。代價：所有 expert 權重都須進顯存(~62GB)，且需 Qwen3VLMoeForConditionalGeneration 類別。

為何 30B-A3B 答案比 8B 簡短/籠統？ 這非 MoE 本質。主因：(1) 兩模型的 instruction-tuning 風格不同(30B 偏精簡)；(2) 都用 greedy 解碼(do_sample=False)；(3) 我們只比了 3 影格，樣本小、屬軼事級觀察。不能推論「MoE 就比較籠統」。

8B 每影格固定 4 題是模型限制嗎？ 不是。是我的 prompt 要求「generate 4 diverse QA pairs」；8B 嚴格照辦，30B 有時多給(3–7 題)。題數由 prompt 決定，非模型規範。

若缺 Summary / Temporal / Trajectory 會怎樣？ 這三類分別測「場景整體理解 / 時間順序 / 動作路徑」。若缺：VQA 會偏向靜態屬性(顏色/位置)，無法評估時序與全局推理；而對 VLA 來說，時序與軌跡理解正是與 action 最相關的維度，缺了會低估 VLM 對操作任務的幫助。本專案 7 類齊全即為涵蓋這些面向。

5 · 案例分析

案例 A — VLM：本專案（Cosmos-R2 VQA LoRA）

影像+問答多模態；凍結 vision、LoRA 調 LLM；label-mask 只對答案算 loss；指標：VQA 準確率（按題型）+ 置換進 VLA 後的 action MSE/MAE。完整數字見專案報告 §6。

案例 B — 純 LLM LoRA（對照）

純文字指令微調（如客服/領域問答）：無 vision/projector、無 image token；資料是純文字對話；指標多為文字準確率/BLEU/人評。相同點：凍結底座、低秩 A·B、label-mask 答案、bf16+grad-ckpt。相異點：VLM 多了視覺編碼、image-token 對齊與更長序列/顯存壓力。

6 · 本專案腳本逐一說明（fine-tune / eval pipeline）

腳本作用關鍵 API
gen_vqa_from_lerobot.py 從 LeRobot v2 影片依 6 個動作階段（approach / grasp / pick / move_to_plate / place / retract）各抽 1 幀，使用 teacher VLM 生成 VQA JSONL；輸出 data.train.jsonl、data.val.jsonl、images/、stats.json AutoModelForImageTextToText、AutoProcessor、cv2.VideoCapture
vqa_dataset.py JSONL → PyTorch Dataset + Collator；將 <image> 佔位字串轉換為結構化 content（[{type:image}, …]），以正確觸發 image token 展開；label masking 將 prompt/影像 token 設為 IGNORE_INDEX=-100，只對 assistant 答案 token 計算 loss processor.apply_chat_template、IGNORE_INDEX=-100
lora_args.py 定義 LoRA 超參數（lora_r、lora_alpha、lora_dropout）與 target 模組（llm 模式：q/k/v/o/gate/up/down proj，可選額外掛 vision encoder）；構建 peft.LoraConfig peft.LoraConfig(task_type="CAUSAL_LM")
train_vlm_lora.py 標準 HF SFT 流程：載入底座 → get_peft_model 插入 LoRA → Trainer.train；訓練結束後儲存 adapter 與 processor Qwen3VLForConditionalGeneration、Trainer、TrainingArguments、get_peft_model
merge_lora.py 將 LoRA adapter 併回底座權重，輸出獨立可推論的 VLM（Product A）；使用 safe_serialization=True 以 safetensors 格式儲存 PeftModel.merge_and_unload()
swap_backbone.py 純 safetensors 操作：將 Product A 的 backbone.model.* 鍵值置換回現有 VLA checkpoint，以 VLA 端的 backbone key 逐一到 merged 對應取值（兩邊都有且形狀相符才置換，merged 多出的層自動略過），保留 action head 不動；輸出「置換後 VLA」（Product B） safetensors.torch.load_file、safetensors.torch.save_file（key 交集 swap）
eval_vlm_vqa.py 對 VLM checkpoint 執行 VQA 準確率評測：按 type 欄位分類統計，答案正規化後做子字串匹配；輸出 per-type 與 Overall 準確率 JSON model.generate、processor.batch_decode、正規化子字串比對
eval_toolcall.py 評測 tool-calling 輸出：解析預測文字與 gold，計算 valid-parse、skills-in-vocab、name-sequence、args-exact 四項指標；使用 ToolCallVLM（來自 serve/model.py）在 GPU 上推論 parse_tool_calls、ToolCallVLM.complete
infer_vlm.py 單張影像 + 問題的快速推論腳本（Product A 獨立使用，無 GR00T dependency）；命令列直接呼叫 Qwen3VLForConditionalGeneration、AutoProcessor、model.generate
serve/model.py 封裝 ToolCallVLM 類別：載入 merged VLM 一次並重複使用；解析 OpenAI 格式的 multimodal image_url（data-URI 或路徑），插入 tool-describing system prompt，回傳 {text, calls} Qwen3VLForConditionalGeneration、build_tool_system、parse_tool_calls（FastAPI 層在 serve/openai_app.py）
train_vlm_lora.py 是 transformers 標準 SFT 流程嗎？與 GR00T N1.7 訓練有何不同？
train_vlm_lora.py 是 transformers 標準 SFT 流程（AutoProcessor + Qwen3VLForConditionalGeneration + peft.get_peft_model + Trainer / TrainingArguments），但綁定 Qwen3-VL 類別與其 proj 名稱；換 VLM 家族需改 AutoModelForImageTextToText 子類與 LoRA target 模組名稱。

與 GR00T N1.7 訓練本質不同——後者使用 GR00T 自家 trainer，訓練對象是整個 VLA（backbone + action adapter + DiT），loss 為 flow-matching action loss，非 HF Trainer、非 CausalLM token loss。本流程只訓練 VLM 的語言理解能力（VQA），action head 完全獨立，最終透過 swap_backbone.py 以 safetensors 鍵值置換的方式注入 VLA。
