LLM 與 VLM 微調指南

從一般大型語言模型（LLM）到視覺語言模型（VLM）的微調方法、挑戰與解法；以本專案（Cosmos-Reason2-2B VQA LoRA）為實例。含動畫資料流圖。

逐題精準回答

#問題答案
1 vlln 是什麼？vl_self_attention 算 VLM 或 DiT？ vlln 是 Vision-Language LayerNorm 的專案命名：在 Gr00tN1d7ActionHead 中實作為 nn.LayerNorm(config.backbone_embedding_dim)，本 checkpoint 是 LayerNorm(2048)。它不是 VLM backbone，而是 action_head.vlln.*。

vl_self_attention 也不是 VLM backbone、也不是 DiT 本體；它是 action head 內、接在 vlln 後面的 4 層 VL feature adapter，key 前綴是 action_head.vl_self_attention.*。DiT 本體在 action_head.model.*，本 checkpoint 為 32 層。
2 Qwen3VLForConditionalGeneration 是 transformers 公定格式嗎？ 不是跨所有 transformer/VLM 的公定類別名稱；它是 Hugging Face Transformers 對 Qwen3-VL 架構提供的模型 class。公定/通用的是 HF 介面慣例：PreTrainedModel、from_pretrained()、save_pretrained()、generate()、AutoProcessor、以及較通用的 AutoModelForImageTextToText。換成 LLaVA、Idefics、Gemma、Mllama 等 VLM，通常會換成該家族自己的 class 或 AutoModel 對應類別。
3 為什麼程式碼預設 select_layer=12，但 checkpoint 是 16？ configs/model/gr00t_n1d7.py 的 12 只是 dataclass fallback/default。實際訓練與儲存 checkpoint 時，訓練 config 可以覆寫它。本例 checkpoint-150000/config.json 明確寫 select_layer=16，而 safetensors key 掃描也只看到 backbone.model.model.language_model.layers.0..15。所以「程式預設 12」不等於「所有訓練產物都用 12」；部署 checkpoint 以其保存的 config.json 為準。
4 train_vlm_lora.py 是標準 transformers 流程嗎？所有 Qwen-VL 都能用嗎？其他架構怎麼做？ 它是標準 HF/PEFT SFT 配方的本專案實作：AutoProcessor + Qwen3VLForConditionalGeneration + get_peft_model(LoraConfig) + Trainer/TrainingArguments。但它不是所有 VLM 零修改通用腳本。目前程式碼綁定 Qwen3-VL class，LoRA target 也假設 Qwen 類模組名：q_proj/k_proj/v_proj/o_proj/gate_proj/up_proj/down_proj。

若是同樣 Qwen3-VL / Cosmos-Reason2-2B 相容架構，通常可直接用。若是其他架構，標準範本是：換成該模型的 AutoModelForImageTextToText 或專屬 class、確認 processor/chat template、重選 target_modules、保持 label masking 與 Trainer 流程。也就是「流程標準」，但「模型 class 與 target module 名稱要依架構調整」。
5 為什麼說第 16–27 層 delta 被丟棄？不能整個 28 層放回去嗎？ 獨立 Cosmos-R2 LoRA 的確是在完整 28 層 VLM 上訓練，merged VLM 也有 LLM layers 0..27。但 GR00T N1.7 的 VLA backbone 在初始化時執行 while len(language_model.layers) > select_layer: pop(-1)，本 checkpoint 的 select_layer=16，所以 VLA checkpoint 只存在 layers 0..15 的目的 key。

swap_backbone.py 是「以 VLA checkpoint 的 key 為主」逐一找 merged VLM 對應 tensor；VLA 沒有 layers 16–27 的 key，就沒有地方可寫。硬把 28 層都放進去會改變模型結構、forward 計算量、hidden-state 分佈，也會讓已訓練好的 action head 接到不同層語意的特徵；那不是 swap，而是要重建/重訓 VLA 設計。
6 為什麼需要 merge？底座是什麼？ LoRA 訓練輸出的 checkpoint-2000/adapter_model.safetensors 只保存低秩 delta（A/B 矩陣），不是完整 VLM。底座就是原始 pretrained base model，例如 nvidia/Cosmos-Reason2-2B。

未 merge 時，推論需要「底座 + adapter」一起載入；merge 會把 ΔW = B·A·scale 寫回底座權重，輸出完整 standalone VLM（Product A）。這樣 Product A 可用普通 Qwen3VLForConditionalGeneration.from_pretrained() 載入，也可作為 swap_backbone.py 的來源。PEFT 官方也明確說 merge_and_unload 用於把 adapter layers 合併回 base model，以便當 standalone model 使用。
7 safetensors 就是 VLM 參數檔嗎？swap/merge 是針對此檔案置換嗎？ .safetensors 只是安全的 tensor 儲存格式，不自動等於「完整 VLM」。要看它所在目錄與 key：

• Isaac-GR00T-VLM/artifacts/cosmos_r2_lora_6phase/checkpoint-2000/adapter_model.safetensors：392 個 LoRA adapter tensor，key 如 base_model.model.model.language_model.layers.0...lora_A/B.weight，不是完整 VLM。
• artifacts/checkpoints/gr00t/.../checkpoint-150000/model-00001-of-00003.safetensors 等：GR00T VLA sharded checkpoint，含 backbone.model.* 與 action_head.*。
• merged Product A：.../Cosmos-Reason2-2B-lora-merged-6phase/model.safetensors，是完整 standalone VLM 權重。

merge_lora.py 載入底座 + adapter，輸出新的 merged safetensors；swap_backbone.py 讀 VLA safetensors 與 merged VLM safetensors，只替換 VLA 中 backbone.model.* 且形狀相符的 tensor，再輸出新的 VLA safetensors。
8 「凍結 adapter/DiT 需要承受小幅 VL 特徵偏移」具體是什麼偏移？ VLA action head 原本是用某個 backbone 特徵分佈訓練的：特徵張量形狀仍是 [B, seq_len, 2048]，但 swap 後，vision/embedding/LLM 前 16 層權重經 VQA LoRA 改過，會讓每個 token 的 hidden vector 數值分佈改變。例如：向量均值/方差、attention 後的 token 關係、物體/顏色/任務文字 token 的語意方向、不同層 feature scale 都可能小幅偏移。

由於 action_head.vlln、4 層 vl_self_attention、32 層 DiT、state/action encoders/decoder 都保持原權重，它們必須仍能把這批稍微改變的 VL features 映射到正確 action。若偏移小，open-loop action MSE/MAE 可能持平或改善；若偏移大，DiT 的 cross-attention 會讀到不同語意/尺度的條件特徵，動作誤差會上升。這就是 swap 後必須跑 VLA action eval 的原因。

1 · LLM 微調：方法總覽

方法更新什麼成本/顯存適用
Full fine-tune全部權重最高（需存全部梯度+優化器狀態）資料多、要最大改變
LoRA凍結原權重，學低秩旁路 A·B低（僅 ~0.1–1% 參數）多數領域微調（本專案）
QLoRA4-bit 量化底座 + LoRA最低（單卡可調大模型）顯存受限
Adapter每層插入小瓶頸模組低多任務切換
Prompt / Prefix tuning只學軟提示向量極低輕量任務適配

LoRA 原理（動畫）：凍結 W，只學低秩 A·B

只有低秩 A、B（綠）可訓練，原始 W 凍結。合併時 W' = W + B·A，得到與原模型同形狀的權重 → 可直接「置換」回原架構（本專案 swap 的基礎）。本專案：r=16, α=32, 僅 0.81% 參數可訓練。

標準步驟

  1. 資料準備：整理成指令/對話格式；切 train/val；定義標籤遮罩（只對「答案」算 loss）。
  2. 選 target 模組：LoRA 通常掛在注意力 q/k/v/o 與 MLP gate/up/down proj。
  3. 超參數：r（秩）、α（縮放）、dropout、lr（1e-4 常見）、batch×grad-accum、warmup、steps。
  4. 訓練：bf16 + gradient checkpointing 省顯存；監看 loss 與 grad_norm。
  5. 評估：held-out 準確率／任務指標；避免過擬合（小資料用小 r + 早停）。

4 · Dense vs MoE（非 dense）

MoE 路由（動畫）：每 token 只啟動少數 expert

Router 為每個 token 挑選少數 expert（綠色啟動）。Qwen3-VL-30B-A3B：31B 總參數，每 token ~3B 啟動 → 算力像小模型、品質像大模型；但全部 expert 都要載進顯存（~62GB）。dense（8B）則每 token 走全部參數、~17GB、載入單純。

MoE 深入：router、expert、協同運作（你問的）

問題說明
router 是內建組件嗎？是。每個 MoE 層內建一個 gating(router) 線性層，對每個 token 算出對各 expert 的分數(softmax)，選出 top-k。它與 expert 一起被訓練。
expert 是什麼？每個 expert 就是一個獨立的前饋網路(FFN)；同一層有 N 個並行 expert。訓練中各 expert 會自然特化到不同的輸入模式/領域。
如何協同？router 為每個 token 選 top-k expert → 只計算那 k 個 → 用 router 分數加權合併它們的輸出；其餘 expert 對該 token 不計算（這就是省算力的來源）。
微調的作用/優勢固定算力下提供更大的「總容量」(31B)，但每 token 僅 ~3B 參與計算 → 推論快、表達力強。代價：所有 expert 權重都須進顯存(~62GB)，且需 Qwen3VLMoeForConditionalGeneration 類別。

為何 30B-A3B 答案比 8B 簡短/籠統？ 這非 MoE 本質。主因：(1) 兩模型的 instruction-tuning 風格不同(30B 偏精簡)；(2) 都用 greedy 解碼(do_sample=False)；(3) 我們只比了 3 影格，樣本小、屬軼事級觀察。不能推論「MoE 就比較籠統」。

8B 每影格固定 4 題是模型限制嗎？ 不是。是我的 prompt 要求「generate 4 diverse QA pairs」；8B 嚴格照辦，30B 有時多給(3–7 題)。題數由 prompt 決定，非模型規範。

若缺 Summary / Temporal / Trajectory 會怎樣？ 這三類分別測「場景整體理解 / 時間順序 / 動作路徑」。若缺：VQA 會偏向靜態屬性(顏色/位置)，無法評估時序與全局推理；而對 VLA 來說，時序與軌跡理解正是與 action 最相關的維度，缺了會低估 VLM 對操作任務的幫助。本專案 7 類齊全即為涵蓋這些面向。

train_vlm_lora.py 是 transformers 標準 SFT 流程嗎？與 GR00T N1.7 訓練有何不同？
train_vlm_lora.py 是 transformers 標準 SFT 流程（AutoProcessor + Qwen3VLForConditionalGeneration + peft.get_peft_model + Trainer / TrainingArguments），但綁定 Qwen3-VL 類別與其 proj 名稱；換 VLM 家族需改 AutoModelForImageTextToText 子類與 LoRA target 模組名稱。

與 GR00T N1.7 訓練本質不同——後者使用 GR00T 自家 trainer，訓練對象是整個 VLA（backbone + action adapter + DiT），loss 為 flow-matching action loss，非 HF Trainer、非 CausalLM token loss。本流程只訓練 VLM 的語言理解能力（VQA），action head 完全獨立，最終透過 swap_backbone.py 以 safetensors 鍵值置換的方式注入 VLA。

你問的幾個概念（CausalLM loss / HF Trainer / 26 維）

問題說明
什麼是 CausalLM loss？和 HF Trainer 有何不同？ 兩者是不同層次、非替代關係：
• CausalLM loss = 自回歸語言模型「預測下一個 token」的交叉熵（next-token cross-entropy），是訓練目標(loss)。把 labels 傳進 Qwen3VLForConditionalGeneration 時模型內部就算它；本專案再加 label-mask，只對 assistant 答案 token 計入（prompt／影像 token 設 -100 略過）。
• HF Trainer = 訓練的流程引擎(loop)：負責 batching、grad-accum、optimizer step、lr schedule、logging、存檔、分散式等，不決定 loss（loss 由 model 回傳）。
所以「用 HF Trainer」與「用 CausalLM loss」是正交的：Trainer 是外圈迴圈，CausalLM loss 是裡面的目標函數。
GR00T 若也微調 backbone，訓練方式也不同嗎？ 仍然不同。即使 GR00T 解凍 backbone（tune_llm=True / tune_top_llm_layers>0 / 對 backbone 掛 LoRA），它不是用 CausalLM／VQA loss、也不是用 HF Trainer。backbone 的梯度是從 flow-matching action loss 經 DiT／adapter 反傳回來——目的是讓 backbone 產生「更利於預測動作的特徵」，而非更會生成文字。差異不只在「動不動 backbone」，而在目標函數與訓練迴圈本身：GR00T＝自家 trainer＋flow-matching action loss；本流程＝HF Trainer＋CausalLM 文字 loss。
為什麼 flow-matching action loss 是 26 維，不是 32？ 26 是 OpenArm O6 這個 embodiment 的真實動作維度：雙臂各 7-DoF（14）＋雙靈巧手各 6-DoF（12）＝ 26（見 dataset 的 action.shape=[26]、observation.state.shape=[26]）。模型為支援多體系，把動作補零到 max_action_dim=132 的統一槽位，並以 action_mask 只對真實 26 維算 loss。32 不是動作維度——那是別處的數字（DiT 的 attention heads＝32、max_num_embodiments＝32）。「26 維」＝真實自由度、「32」＝注意力頭／體系數，兩者無關。
