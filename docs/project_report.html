4 · Teacher 比較：Qwen3-VL-30B-A3B (MoE) vs Qwen3-VL-8B-Instruct (dense)

什麼是 dense / 非 dense（MoE）？

Dense（如 Qwen3-VL-8B-Instruct）MoE / 非 dense（如 Qwen3-VL-30B-A3B）
FFN 結構每個 transformer block 一個前饋網路，每個 token 走過全部參數每個 block 多個 expert FFN + router(gating)，每個 token 只啟動少數 expert
參數8B 全部啟動31B 總量、每 token 僅 ~3B 啟動（A3B）
transformers 類別Qwen3VLForConditionalGenerationQwen3VLMoeForConditionalGeneration
載入~17GB、單一類別、直接載入所有 expert 都要進顯存 ~62GB（bf16，13 shards）；「快」指算力非記憶體；需 device_map=auto，建議 flash-attn
載入複雜點無需確認 transformers 含 MoE 類別（4.57.3 已含）；fp32 會 ~124GB 爆 80GB → 必須 bf16

同一影格下兩者產生的 QA（實測，task：place the can on the orange plate）

觀察：30B-A3B (MoE) 題型覆蓋較廣（某些影格產 7 題，含 Summary/Temporal/Trajectory），但答案較簡短、推理偏籠統（如 reasoning：「to complete a task」）；8B (dense) 每影格穩定 4 題、格式一致、推理較具體落地（「To place it on the orange plate as per the task」）。啟示：對「結構化抽取」這類任務，較大的 MoE 未必一致更好；dense 8B 在格式穩定與具體性上反而占優。本專案以 30B-A3B 為主 teacher 取其覆蓋廣度，template 提供高信度骨幹。

7 · 產物 A 的獨立應用

產物 A 是完整的 Qwen3-VL checkpoint，不依賴 GR00T，純 transformers 即可載入做 VQA／場景理解。smoke 階段驗證：對橘盤影像問「What color is the plate?」→ 模型答「Orange.」。可 push 到 HF Hub 或複製到任意推論環境部署。

8 · 結論與未來改進

  • 可行性已證：獨立對 Cosmos-R2 做 VQA LoRA → 合併 standalone VLM → 無損置換進 VLA（493/493），全程不動 n1d7。
  • side-effect：swap 後 VLM 特徵分佈改變、action head 未重新適配；PDF 經驗顯示 action 多半「維持」而非提升——VLM 理解力提升不必然轉成 action 提升。未來可在 swap 後對 action head 輕量再訓練。
  • 資料多樣性：本資料集僅 2 任務/單相機，過擬合風險高；未來擴充多資料集（real+sim、head/wrist 多相機）。
  • teacher 選擇：30B-A3B 覆蓋廣、8B 具體穩定；可混用或加人工校正提升 QA 品質。
